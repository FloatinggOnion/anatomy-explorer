    Update tauri.conf.json to finalize build, security, and permission configuration:
    
    **Build configuration:**
    ```json
    "build": {
      "beforeDevCommand": "pnpm dev",
      "beforeBuildCommand": "pnpm build",
      "devUrl": "http://localhost:5173",
      "frontendDist": "../dist"
    }
    ```
    
    **Security (CSP):**
    ```json
    "security": {
      "csp": {
        "default-src": ["'self'", "data:", "blob:"],
        "script-src": ["'self'"],
        "style-src": ["'self'", "'unsafe-inline'"],
        "img-src": ["'self'", "data:", "blob:"],
        "connect-src": ["'self'", "http://localhost:5173"],
        "font-src": ["'self'"],
        "worker-src": ["'self'", "blob:"],
        "child-src": ["blob:"]
      }
    }
    ```
    
    Explanation:
    - `data:` and `blob:` in default-src: Allow MediaPipe WASM and other blob-based assets
    - `'unsafe-inline'` in style-src: Required by Tailwind CSS
    - `worker-src: ['self', 'blob:']`: Allow MediaPipe to spawn web workers from blobs
    - `http://localhost:5173` in connect-src: Allow dev server HMR in dev mode
    
    **Window configuration:**
    ```json
    "windows": [
      {
        "title": "AR Anatomy Explorer",
        "width": 1200,
        "height": 800,
        "resizable": true,
        "fullscreen": false,
        "transparent": false
      }
    ]
    ```
    
    **Capabilities (required for Tauri permissions):**
    Add capability for camera permission. Tauri v2 uses a capabilities-based permission system. Ensure `tauri.conf.json` includes:
    ```json
    "tauri": {
      "...": "...",
      "allowlist": {
        "window": {
          "all": true
        },
        "shell": {
          "open": true,
          "execute": false
        }
      }
    }
    ```
    
    Note: Camera permission in Tauri is handled via native OS prompt, not Tauri configuration. The pre-permission screen (D-01) in the UI mitigates the Windows one-way door issue.
    
    **Testing notes:**
    - In dev mode, `devUrl: "http://localhost:5173"` points to Vite dev server for HMR
    - In production, `frontendDist: "../dist"` points to built web assets
    - Both modes use same React code and permission flow

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS kernel ← App | App requests camera permission; OS enforces access control |
| Tauri webview ← OS | WebView2 enforces CSP and origin policy |
| React code ← Build system | Vite bundles and minifies; no untrusted input injected at build time |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-PLAN-04-A | Spoofing | Pre-permission screen | mitigate | Explanation text is honest ("This app needs camera access"). No misleading language. Button is clearly labeled. |
| T-01-PLAN-04-B | Tampering | Tauri CSP headers | mitigate | CSP locked in tauri.conf.json. No inline scripts allowed (script-src: 'self'). Vite-bundled JS is trusted (build-time only). |
| T-01-PLAN-04-C | Repudiation | Camera access logging | accept | Phase 1 does not log camera access. Logging can be added in Phase 2 if audit trail is required. |
| T-01-PLAN-04-D | Denial of Service | Large model file | mitigate | Enforced 5MB budget for skeleton.glb. App remains responsive even if model loads slowly. |
| T-01-PLAN-04-E | Information Disclosure | WebView2 environment | accept | Tauri webview is isolated from OS filesystem (except for explicitly allowed APIs). Camera stream is not persisted. No PII exposed. |
| T-01-PLAN-04-F | Elevation of Privilege | Tauri allowlist | mitigate | Capabilities are minimal (window management only). No file I/O, shell execution, or system API access in Phase 1. Prevent privilege escalation via narrow allowlist. |

</threat_model>
