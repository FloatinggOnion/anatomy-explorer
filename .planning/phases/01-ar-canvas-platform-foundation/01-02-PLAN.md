<objective>
Create the webcam input pipeline: getUserMedia() lifecycle management, pre-permission screen flow (D-01), permission denial fallback (D-03), and Zustand state tracking. Users see permission explanation before browser prompt, see live feed on grant, see checkerboard on deny, and can use the app in mouse-only mode either way.

Purpose: Lock the permission flow early — this is a one-way door on Windows (D-02 concern). Get the UX right and test thoroughly before Phase 2 adds hand tracking.

Output: A React component system that manages webcam access safely, with clear pre-permission flow and graceful fallback to mouse-only mode.
</objective>

  <action>
    Create src/hooks/useWebcam.ts with:
    
    **Hook signature:**
    `export function useWebcam() { ... return { permissionState, error, videoRef, startCamera } }`
    
    **Behavior:**
    1. Initialize permissionState from Zustand store
    2. Maintain local videoRef using useRef<HTMLVideoElement>(null)
    3. Implement startCamera() function:
       - Set permissionState to 'pending'
       - Call navigator.mediaDevices.getUserMedia({ video: { width: 640, height: 480 } })
       - On success: set permissionState to 'granted', attach stream to videoRef.current.srcObject
       - On error (denied, timeout, etc.): set permissionState to 'denied', set error message
    4. useEffect to clean up: stop all tracks on unmount or permission change
    
    **Error handling:**
    - NotAllowedError (user clicked Deny) → set permissionState to 'denied'
    - NotFoundError (no camera device) → set permissionState to 'denied', error "No camera found"
    - Other errors → set permissionState to 'denied', error message from exception
    
    **Do NOT request permission on mount.** Only when startCamera() is called (user clicks button).
    
    videoRef is stored in Zustand for later retrieval by hand tracking code in Phase 2.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser ← User gesture | Permission prompt controlled by user; pre-permission explanation builds trust |
| App ← Webcam stream | getUserMedia() returns MediaStream controlled by OS; user must grant permission |
| Tauri webview ← Webcam | WebView2 on Windows may have permission persistence — must be tested |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-PLAN-02-A | Spoofing | Pre-permission screen | mitigate | Explanation text is clear and non-deceptive. "Start Camera" button clearly labeled (not "Agree to all"). No auto-start. |
| T-01-PLAN-02-B | Tampering | WebcamProvider video stream | accept | MediaStream from getUserMedia is controlled by OS kernel. App can read but not inject frames. No attacker can fake the webcam feed in the browser. |
| T-01-PLAN-02-C | Information Disclosure | Webcam access to sensitive background | mitigate | No processing of camera feed in this plan — just display. Hand tracking and gesture analysis come in Phase 2. Users are informed via pre-permission screen that camera is accessed. |
| T-01-PLAN-02-D | Denial of Service | Malformed video stream | accept | getUserMedia returns valid MediaStream or throws error. App handles error gracefully (checkerboard fallback, mouse-only mode). No way for malformed stream to crash renderer. |

</threat_model>
